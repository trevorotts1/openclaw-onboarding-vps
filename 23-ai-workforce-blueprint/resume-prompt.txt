[WORKFORCE-RESUME]

You are receiving this message from your own cron — it is NOT from the owner and NOT a new user request. Treat it as an internal build-resume invocation.

PROTOCOL (per Skill 23 INSTRUCTIONS.md → "Post-Interview Handoff Protocol"):

1. Read `/data/.openclaw/workspace/.workforce-build-state.json` (Mac: `~/.openclaw/workspace/.workforce-build-state.json`).
2. Identify departments where `status == "pending"` or `status == "failed"` or (`status == "building"` AND `lastAttemptAt` older than 15 minutes — that means the prior build session died).
3. For each such department, BEFORE starting work:
   a. Flip its `status` to `"building"` and set `lastAttemptAt` to now (UTC ISO-8601).
   b. Write the state file back to disk atomically (tmp + rename).
4. Generate the department per the standard Skill 23 build (read `workforce-interview-answers.md` or the equivalent v2.0/v10.14.14 dossier — DO NOT re-interview).
5. After EACH department finishes (or fails), update its entry:
   - `status` → `"done"` (success) or `"failed"` (with `failureReason`)
   - `completedAt` → now
   - `filesPresent` → list of materialized files
   - Persist state file immediately
6. After ALL departments are `done`, write `buildCompletedAt` to state file AND send ONE consolidated status message to `ownerChat` (the owner gets one ping, not many).
7. Do NOT message the owner during the resume — they only hear from you when the whole build is done.
8. Do NOT increment `resumeAttempts` — the resume script already did that before invoking you. You only write department state.

Spawn parallel sub-agents (per the "Generation Orchestration" stages 3-4 of INSTRUCTIONS.md) to move fast — up to 10 dept agents concurrently, up to 50 role agents concurrently.

If you encounter ANY error you cannot recover from after 3 retries, mark that department `"failed"` with a `failureReason` and continue with the others. Don't block the whole build on one stuck department.
