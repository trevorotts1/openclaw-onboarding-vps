# PA-23-02: Milestone Map Build

**Department:** Goal Setter (Specialist 23)
**SOP ID:** PA-23-02
**Phase:** DMAIC Section 3 — Analyze
**Version:** 1.0
**Token:** {{TOKEN}}

---

## Purpose

To take each approved goal from the Definition Sprint (PA-23-01) and break it into a sequenced, time-bound milestone map. This SOP ensures {{TOKEN}} never faces a big goal alone — every destination has a clear path with visible checkpoints.

## Prerequisites

- Completed Goal Definition Sprint (PA-23-01) with 3-5 SMART goals
- Calendar access for the full goal period
- Understanding of {{TOKEN}}'s team capacity and seasonal rhythms

## Procedure

### Step 1: Goal Decomposition (30 min per goal)

Take one goal at a time. Start from the end state and work backward. For each goal, ask: "What must be true 30 days before the deadline for this to succeed?" Then: "What must be true 60 days before?" Continue until you reach today.

Write each "must be true" statement as a milestone. A proper milestone has three qualities: it's binary (done or not done — no 70%), it's independently verifiable (someone who knows nothing about the project can confirm completion), and it has a calendar date.

Example: If {{TOKEN}}'s goal is "Launch membership site by December 15," backward milestones might be:
- **Nov 15:** Payment processing live with test transaction
- **Oct 15:** All core content uploaded and formatted
- **Sep 15:** Platform selected and developer engaged
- **Aug 15:** Content outline complete, platform evaluation matrix done

### Step 2: Dependency Mapping (15 min per goal)

Draw connections between milestones — within the same goal and across goals. Mark each milestone as:
- **Independent:** Can start anytime
- **Dependent on [X]:** Blocked until another milestone completes
- **Parallel with [Y]:** Can run simultaneously with another milestone

Look for bottlenecks. If three dependent milestones all feed into one person or one week, flag it immediately. {{TOKEN}} needs to know where the constraints are before they become crises.

### Step 3: Capacity Check (20 min)

Overlay the milestone map onto {{TOKEN}}'s actual calendar. Account for:
- Known busy periods (launches, events, holidays)
- Team availability and vacation windows
- {{TOKEN}}'s personal energy patterns (morning person? weekly sprinter?)
- Buffer: add 25% to every time estimate — the planning fallacy is real

If any month shows more than three major milestones, it's overloaded. Resequence or descope before moving forward.

### Step 4: Early Warning System (15 min)

For the three most critical milestones, define a "yellow flag" trigger — a measurable signal that shows we're trending off track BEFORE the milestone is missed. Example: if the milestone is "100 sales calls by November 1," the yellow flag is "fewer than 25 calls by October 1."

Document each trigger with the person responsible for monitoring it and the action {{TOKEN}} takes when it fires.

## Output

A milestone map document containing:
- Visual timeline (Gantt-style or simple calendar view)
- Per-goal milestone list with dates, dependencies, and yellow-flag triggers
- Capacity assessment with flagged overload months

## Quality Gate

- [ ] Every goal has 4-7 milestones with hard dates
- [ ] All dependencies are explicitly mapped
- [ ] No month exceeds three major milestones without a documented mitigation
- [ ] Each critical milestone has a yellow-flag trigger with owner

---

*This SOP is Section 3 (Analyze) of the DMAIC cycle. It examines the structural path from current state to goal achievement, identifying constraints and failure points before execution begins.*

## Escalation

| Signal | Route To |
|---|---|
| Goal decomposition reveals an impossible timeline | Specialist 21 (Clarity Specialist), PA-21-03 — reset priorities |
| Three or more milestones feed into one person/one week | Flag immediately; Specialist 17 (The Challenger) |
| Capacity check shows overloaded months without mitigation | Descope or re-sequence; Specialist 23, PA-23-04 |
| Dependency map reveals circular dependencies | Operator — structural plan flaw |

## Definition of Done

{{TOKEN}} has 4–7 dated milestones per goal with all dependencies mapped, a calendar-validated capacity assessment, and yellow-flag triggers on every critical milestone.

## Tone & Persona Note

Architect, not cheerleader. The map is a tool {{TOKEN}} trusts because it is honest — it shows bottlenecks, not just sunshine. "Let's see where it actually gets tight" invites truth without fear. Faith-respectful; culturally fluent for Black women professionals.
