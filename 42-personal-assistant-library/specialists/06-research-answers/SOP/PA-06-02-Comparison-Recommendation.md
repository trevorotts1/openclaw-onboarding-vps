# SOP PA-06-02 -- Comparison & Recommendation

**SOP ID:** PA-06-02
**Department:** 06 -- Research & Answers
**Title:** Comparison & Recommendation
**Version:** 1.0
**Last Updated:** 2026-06-02
**Estimated Time:** 15–45 minutes

---

## Purpose

When {{TOKEN}} is choosing between options -- tools, strategies, vendors, platforms -- this SOP delivers a structured comparison table, a weighted analysis, and a clear recommendation so {{TOKEN}} can decide with confidence instead of drowning in browser tabs.

---

## Section 3 -- DMAIC Framework

### 3.1 Define

**Trigger:** {{TOKEN}} presents two or more options and needs help choosing.

**Examples:**
- "Should I use Zoom or Google Meet for the webinar?"
- "Compare Notion vs. ClickUp for our project tracking."
- "Which email provider is better for cold outreach: Instantly or SmartLead?"

**Goal:** Deliver a side-by-side comparison with a recommendation and the reasoning behind it. {{TOKEN}} should understand WHY, not just WHAT.

**Success looks like:** {{TOKEN}} reads the comparison and makes a decision without asking follow-up questions about the options.

### 3.2 Measure

| Metric | Target |
|--------|--------|
| Options compared | All options {{TOKEN}} mentioned, plus any obvious alternatives |
| Criteria depth | Minimum 4 dimensions (cost, features, ease of use, support) |
| Recommendation confidence | Stated explicitly (Strong / Moderate / Tentative) |
| Bias disclosure | Any affiliation or preference disclosed upfront |

### 3.3 Analyze

**Steps -- the Structured Comparison Path:**

1. **Confirm the use case** -- What is {{TOKEN}} actually trying to accomplish? "Which tool is better" depends entirely on the job. Clarify before comparing.
2. **List the options** -- Include the ones {{TOKEN}} named, plus one or two obvious alternatives they may have missed. Flag that you added them.
3. **Define comparison criteria** -- Pick 4–6 dimensions that matter for THIS decision. Common: price, features, ease of use, integrations, support quality, scalability.
4. **Research each option** against each criterion. Use official sites, user reviews, API docs, and pricing pages. Note where information is incomplete.
5. **Score and weight** -- Assign a simple score per criterion (1–5). Weight the criteria by importance to {{TOKEN}}'s use case (e.g., price might be 2× for a bootstrapped project).
6. **Write the recommendation** -- Lead with the winner and why. Then present the comparison table. Then add caveats and alternatives.

**Output format:**

```
**Recommendation:** [Winner] -- [one-sentence reason]

**Comparison Table:**
| Criterion | Weight | Option A | Option B | Option C |
|-----------|--------|----------|----------|----------|
| [each row] | | | | |

**Score:** A: X/25 · B: Y/25 · C: Z/25

**Bottom Line:** [2–3 sentences wrapping up]
**Confidence:** [Strong / Moderate / Tentative -- with note]
```

### 3.4 Improve

- After {{TOKEN}} makes the final call, note which criteria actually drove the decision. Adjust weighting for future comparisons.
- If a recommended option is later rejected or fails, log the lesson so it informs future research.
- Build a running catalog of tools {{TOKEN}} has evaluated so you do not re-research from scratch.

### 3.5 Control

- Always disclose if you found a significant limitation that could be a dealbreaker, even if it hurts the recommendation.
- If pricing data is time-sensitive (trial, promo, "call for quote"), flag the date you checked it.
- Never recommend a tool you cannot find current pricing for without noting that gap.

## Warm PA Voice Notes

"After comparing the three, I'd recommend Zoom for the webinar -- it handles large audiences more reliably and you already have the Pro account. Google Meet is simpler if the group is under 100, but Zoom gives you breakout rooms and recording controls you'll actually use. Here's the full breakdown..."

Friendly, decisive, transparent about the reasoning. {{TOKEN}} should feel like they just got a briefing from a trusted team member.

## CTQ (Critical to Quality) — Binary Checks
- [ ] Use case confirmed before comparing options — "what job is this solving?" answered.
- [ ] Minimum 4 comparison criteria applied (cost, features, ease of use, support).
- [ ] At least one alternative {{OWNER_NAME}} did not mention surfaced and flagged.
- [ ] Recommendation confidence level stated explicitly (Strong / Moderate / Tentative).
- [ ] Affiliations, biases, or information gaps disclosed upfront.

## Metrics
1. **Decision Closure Rate** — Percentage of comparisons where {{OWNER_NAME}} makes a decision without follow-up questions. Target: ≥80%.
2. **Criteria Depth** — Average number of dimensions compared. Target: ≥4.
3. **Post-Decision Satisfaction** — {{OWNER_NAME}} rates satisfaction with recommendation at 30-day retro. Target: ≥4/5.

## Escalation
- If options are technically complex and need vendor due diligence → partner with **06 (Research & Answers) SOP PA-06-03**.
- If the comparison reveals a strategic pivot opportunity → partner with **07 (Brainstorming & Ideation) SOP PA-07-04**.
- If {{OWNER_NAME}} is stuck between options and needs decision coaching → partner with **08 (My Coach) SOP PA-08-04**.

## Definition of Done
Structured comparison delivered with weighted recommendation, all options scored, and confidence level stated — {{OWNER_NAME}} can decide without additional research.

## Tone & Persona Note
Friendly, decisive, transparent about the reasoning. {{OWNER_NAME}} should feel like they just got a briefing from a trusted team member — not a sales pitch. Lead with the recommendation, then show the work. Never hide a limitation that could be a dealbreaker.
