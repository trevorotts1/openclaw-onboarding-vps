# Notion Client-Doc Standard

The **client Notion setup doc** (the "Client Reference Sheet", generated by
`scripts/21-generate-client-reference-sheet.sh` and delivered into the CLIENT's own account — Notion →
Google Docs → text) is the single doc a client opens to wire OpenClaw into their GHL (Convert and Flow)
account and to understand what was set up. This document is the single standard for that doc's STRUCTURE
and the ORDERED CHECKLIST of items that MUST appear in it.

> **Why structure matters.** A live client was once stranded by a reference sheet with neither a bearer
> token nor a copyable Raw Body — there was nothing to paste into GHL. Another opened "their" doc and saw
> someone else's name (a universal-skill leak). This standard makes the doc bulletproof and machine-checked
> so neither recurs.

---

## 0. EVERY CLIENT NOTION SETUP DOC MUST INCLUDE ALL OF THE FOLLOWING, IN THIS ORDER (BINDING — NON-NEGOTIABLE)

> **STANDARDIZATION CONTRACT.** Two different installs of this skill must produce structurally identical
> client docs — same sections, same order, only the substituted values (the hostname, the hook id, the
> token, the business name) change. No reordering, no dropping a section. If ANY one is missing or
> out-of-order, the doc is INVALID and the install is not complete. This §0 is what
> `scripts/qc-notion-doc-standard.sh` asserts is documented here, and what `scripts/qc-reference-sheet.sh`
> machine-enforces against the actually-generated doc (both wired into `scripts/11-run-qc-checklist.sh` + CI).

EVERY CLIENT NOTION SETUP DOC MUST INCLUDE ALL OF THE FOLLOWING, IN THIS ORDER:

1. **🚀 Quick Start FIRST.** The doc LEADS with a literal "🚀 Quick Start" section — the copy-paste values
   and steps — BEFORE any explanation. The client can wire it up without reading the theory.
2. **Webhook URL — its OWN code block.** The exact hook URL (`https://<PUBLIC_HOSTNAME>/hooks/<HOOK_NAME>`)
   in its own copyable fenced code block (no trailing slash).
3. **Authorization — TWO separate code blocks.** Block 1 is exactly `Authorization` (the Header KEY field);
   block 2 is exactly `Bearer <token>` (the Header VALUE field), value-only. **NEVER** a combined
   `Authorization: Bearer <token>` block — the value box holds ONLY `Bearer ` + the token and must NOT
   contain the word "Authorization".
4. **Content-Type — TWO separate code blocks.** Block 1 exactly `Content-Type` (KEY); block 2 exactly
   `application/json` (VALUE). Same two-box reasoning as Authorization.
5. **The FLAT 23-key Raw Body — its OWN ` ```json ` code block.** The canonical FLAT 23-key body (per
   `ghl-raw-body-json-standard.md`), copyable, placeholder-free `messageTemplate`, never nested, never
   under 23 keys.
6. **Build step — tags FIRST + field-by-field manual fill + "Build-with-AI builds the SHAPE only" warning
   + post-build VERIFY.** (a) Create any required tags FIRST (under Settings → Tags) before building; (b)
   field-by-field manual Custom-Webhook fill (Method = POST, URL, Headers, Content-Type, Raw Body); (c) a
   prominent warning that **Build-with-AI builds the SHAPE only** (trigger + filters + an EMPTY webhook) and
   **will not** populate the URL/headers/body — the client MUST fill them by hand; (d) a post-build VERIFY
   checklist (every field non-empty, Trigger / Custom Webhook / Publish, the blank-tag-in-a-`does not
   contain`-filter gotcha).
7. **"Your Communication Playbooks" section.** AFTER the Quick Start, BEFORE the deep how-it-works: where
   playbooks live (`conversation-workflows/` mirrored to Notion), the **"help me build a [purpose]
   playbook"** call-to-action (e.g. "Want another communication playbook? Just ask me!"), the personal
   **trigger word** (voice-assistant style — like Alexa / Hey Siri), the **"I Do / You Do"** process + the
   **~15-30 minute** expectation, and the **brainstorm** ("things to think about" — NOT a 50-question form).
8. **VPS-vs-Mac install considerations.** A section covering the VPS (Hostinger Docker) vs Mac mini
   install differences (host `/docker/<project>/.env` + `force-recreate`; container `secrets/.env`;
   `OPENCLAW_HOOKS_TOKEN`; Mac `openclaw.json` env block + `launchctl kickstart`). Source:
   `references/VPS-VS-MAC-INSTALL.md`.
9. **How-it-works explanation LAST.** The deep "Reference & explanation" section comes AFTER the Quick
   Start, the playbooks section, and the VPS-vs-Mac section — never before.
10. **Every copyable value in its OWN code block.** URL, Authorization key, Bearer value, Content-Type key,
    `application/json` value, and the Raw Body each get their own fenced code block (50+ clients copy each
    field individually).
11. **Delivered to the client via Telegram.** The doc's LINK is sent to the client via the OpenClaw gateway
    (`openclaw message send --channel telegram`). The install is not complete until the client has been
    sent their link — machine-enforced by `scripts/qc-notify-client-doc.sh` (transcript-grep + gated).
12. **UNIVERSAL — no personal/client data.** No real names, real hostnames, real tokens, real location
    ids, real phone/email, or real Telegram ids anywhere in the doc — generic placeholders only
    (`<CLIENT_BUSINESS_NAME>`, `<PUBLIC_HOSTNAME>`, `<HOOKS_TOKEN>`, `<LOCATION_ID>`). Machine-enforced by
    `scripts/qc-no-personal-data.sh` (scans the generated doc too).

These twelve, in this order, are the floor for every client doc this skill emits.

## 1. THE TWO-BLOCK AUTHORIZATION RULE (the most-broken item)

A GHL custom header has a **Key (name) field** and a **Value field** — two separate boxes in the GHL UI.
So the doc renders the Authorization header as **two separate copyable code blocks, never one combined
block**:

- **Block 1** = exactly `Authorization` → paste into the Header **KEY (name)** field.
- **Block 2** = exactly `Bearer <token>` → paste into the Header **VALUE** field. The value block holds
  **ONLY** `Bearer ` + the token; it must **NOT** contain the word `Authorization`. A combined
  `Authorization: Bearer <token>` block is a FAILURE (`scripts/qc-reference-sheet.sh` FAILs the build on
  it).

Do the same split for Content-Type: block 1 `Content-Type`, block 2 `application/json`. The token is the
ACTUAL `hooks.token` resolved at generation time; if it cannot be resolved, the generator emits a
clearly-marked PLACEHOLDER and warns — it never silently omits the section.

## 2. SOURCE OF TRUTH + ENFORCEMENT

- **Generator:** `scripts/21-generate-client-reference-sheet.sh` emits this doc (offline → Layer-3
  markdown when no `openclaw` on PATH; live → resolves the real token/URL).
- **Embedded body:** governed by `ghl-raw-body-json-standard.md` (the FLAT 23-key body), enforced by
  `scripts/qc-23-key-bodies.sh`.
- **Playbooks section:** governed by `communications-playbook-standard.md`.
- **Machine enforcement of the doc structure:** `scripts/qc-reference-sheet.sh` drives the generator in a
  sandbox and FAILs the build if the rendered doc is missing the 🚀 Quick Start (first), the separate
  Authorization KEY + `Bearer` VALUE blocks (and FAILs on a combined `Authorization: Bearer` block), the
  separate Content-Type KEY + value blocks, the copyable ` ```json ` Raw Body, the hook URL, the
  tags-first + manual-fill + "Build with AI will not" warning + post-build verification, the "Your
  Communication Playbooks" section (CTA + trigger-word + I-Do/You-Do + brainstorm), the VPS-vs-Mac section,
  or the after-it Reference & explanation. `scripts/qc-notion-doc-standard.sh` asserts THIS standard
  documents the ordered mandatory list. `scripts/qc-notify-client-doc.sh` enforces the Telegram delivery.
  `scripts/qc-no-personal-data.sh` enforces the universal/no-personal-data rule. All wired into
  `scripts/11-run-qc-checklist.sh` + `.github/workflows/qc-static.yml`.
